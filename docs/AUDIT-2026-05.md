# Thai Use-Case Audit — May 2026
## ผลการตรวจ skill ทั้ง 12 ตัวจากมุมคนใช้งานจริงในไทย

เอกสารฉบับนี้เก็บผลของการตรวจ skill ทั้ง 12 ตัวที่ทำในเดือนพฤษภาคม 2569 (พ.ศ.) เพื่อให้ผู้ดูแลและผู้ contribute ใช้เป็นจุดอ้างอิงในการตามแก้ไขรายการต่างๆ. รายการแต่ละข้อมีหมายเลข (เช่น `A-PDPA-1`) เพื่อให้ PR ถัดไปอ้างถึงได้ตรง.

> This document captures the May 2026 audit of all twelve skills, framed from the perspective of Thai users in real-world contexts. Each finding has a stable identifier (for example `A-PDPA-1`) so that follow-up pull requests can reference specific items rather than relying on prose pointers.

หลักการ:

* รายการที่ระบุว่า **Action** คือสิ่งที่จะแก้ใน PR ถัดไป.
* รายการที่ระบุว่า **Note / Context** คือข้อสังเกตที่อาจกลายเป็น action ในอนาคต แต่ยังไม่ใช่ลำดับความสำคัญสูง.
* การจัดลำดับความสำคัญใช้ผลกระทบต่อ "Thai user" จริง (ไม่ใช่ความน่าสนใจทางเทคนิค) — ตัวเลขใน priority น้อย = สำคัญกว่า.

## รายละเอียดรายตัว / Per-skill findings

### thai-translate

* **A-TR-1 (medium, gap):** ไม่มี **ราชาศัพท์ minimum**. skill ใช้ "ทรงพระเจริญ" ใน register table แต่ไม่ได้อธิบายกฎของกริยาราชาศัพท์ (เสด็จ, ทรง, พระราชทาน, ฯลฯ) ซึ่งจำเป็นเมื่อแปลข้อความที่กล่าวถึงสถาบัน, วัด, หรือสุนทรพจน์เป็นทางการ.
  * *Action:* เพิ่ม subsection "ราชาศัพท์ขั้นต่ำ" 10-15 รายการ + ตัวอย่างใช้ผิด/ถูก.
* **A-TR-2 (low, scope):** ไม่ได้พูดถึงภาษาถิ่น (คำเมือง, อีสาน, ใต้). scope เป็น TH↔EN ไม่ผิด แต่ควรเพิ่มประโยค scope ชัดเจนว่าครอบคลุมเฉพาะ Central Thai.
  * *Action:* เพิ่มประโยค scope หนึ่งบรรทัดที่ส่วน "When to use".

### thai-social-caption

* **A-SOC-1 (medium, currency):** ตารางแพลตฟอร์มยังเขียน **"LINE TIMELINE"** ซึ่งถูกแทนที่ด้วย **LINE Voom** (LINE ค่อยๆ rebrand ตั้งแต่ปี 2565). คนไทยส่วนใหญ่ใช้คำว่า "Voom" หรือ "ฟีด LINE" ในปี 2569 แล้ว.
  * *Action:* เปลี่ยน "LINE TIMELINE" → "LINE Voom" ในตารางและตัวอย่าง.
* **A-SOC-2 (medium, gap):** ไม่ครอบคลุม **Shopee Live / TikTok Live caption conventions** ซึ่งเป็น selling format หลักในไทยตอนนี้. คนละ tone กับ post — ต้องเป็น hook 3 วินาทีแรก, ราคาเด่น, urgency timer.
  * *Action:* เพิ่ม section "Live-stream caption" สั้นๆ 4-5 บรรทัด.

### thai-customer-service

* **A-CS-1 (medium, gap):** ขาด script สำหรับ **"ของไม่ถึง / EMS ล่าช้า / ขนส่งหาย"** — เป็น scenario CS อันดับ 1 ในไทย, มี script แบบ formulaic ที่ผู้ขาย Shopee/Lazada/TikTok Shop ใช้ซ้ำๆ (ตามให้กับ Kerry/Flash/J&T, ขออภัยเรื่องฝน, แจ้งสถานะใหม่).
  * *Action:* เพิ่ม subsection "Shipping issues" พร้อม script ของ 3 carrier หลัก (Kerry, Flash, J&T) + Thailand Post EMS.
* **A-CS-2 (low, gap):** ไม่ได้พูดถึง **LINE OA rich menu** vs **free-text** reply — เป็น UX choice ที่ใช้ใน LINE OA Plus.
  * *Action:* เพิ่มหนึ่งย่อหน้าในส่วน "Channel openers".

### thai-resume

* **A-RES-1 (high, gap):** ไม่กล่าวถึง **ทหาร / military service status**. ผู้ชายไทยอายุ 21+ คาดหวังให้ระบุ "ปลดเป็นทหารกองหนุน" หรือ "ได้รับการยกเว้น" ใน resume สำหรับงานในประเทศ. resume สำหรับงานสากลตัดทิ้ง.
  * *Action:* เพิ่ม subsection "ข้อมูลทหาร" พร้อมกฎเมื่อใส่/ไม่ใส่.
* **A-RES-2 (medium, gap):** ไม่มี template สำหรับ **ใบสมัครงานราชการ** ซึ่งมี structure คงที่ (ก.พ.7, ใบสมัครเฉพาะกระทรวง) ต่างจาก resume ทั่วไป.
  * *Action:* เพิ่ม template หรืออ้างอิง link ของ ก.พ.7.
* **A-RES-3 (low, convention):** ไม่ได้พูดถึง **"ตำแหน่งที่สมัคร"** header — เป็น Thai convention ในการนำหน้า resume.
  * *Action:* เพิ่มในตัวอย่างที่มีอยู่.

### thai-festival-card

* **A-FEST-1 (medium, wording):** "ห้ามมอบของขวัญในงานศพ" เขียนแข็งเกินไป. กฎจริงคือ *ส่งพวงหรีดหรือเงินช่วยงานแทน* — ที่อื่นใน skill เขียนถูกแล้ว แต่ประโยคที่ห้ามเด็ดขาดดูแรงเกินจริง.
  * *Action:* เปลี่ยน "ห้ามมอบของขวัญ" → "ตามธรรมเนียมแล้วไม่มอบของขวัญส่วนตัว — ส่งพวงหรีดหรือเงินช่วยงานแทน".
* **A-FEST-2 (medium, gap):** ขาด template **งานอุปสมบท** — เป็นพิธีสำคัญในชีวิตชายไทยที่ถูกถามให้เขียนการ์ดอวยพรบ่อย.
  * *Action:* เพิ่ม template + บริบทเกี่ยวกับ "ขอแสดงความอนุโมทนา", การจัดงาน.
* **A-FEST-3 (medium, gap):** ขาด template **งานทำบุญขึ้นบ้านใหม่** — common ask, no template.
  * *Action:* เพิ่ม template สั้นๆ.
* *Note (low):* ตัวอย่าง ตรุษจีน ใช้ Hokkien ("ซินเจียยู่อี่ ซินนี้ฮวดไช้"). คน Sino-Thai ส่วนใหญ่เป็น Teochew. Hokkien รู้จักกันดีพอ จึงยังใช้ได้ แต่ note ความหลากหลายไว้ก็ดี.

### thai-government-form

* **A-GOV-1 (high, gap):** ไม่มี **บันทึกข้อความ** (internal memo) — เป็นเอกสารราชการที่เขียนบ่อยที่สุดในชีวิตข้าราชการ. skill ครอบคลุมหนังสือภายนอกแต่ไม่มีรูปแบบของบันทึกข้อความ.
  * *Action:* เพิ่ม section "บันทึกข้อความ" พร้อม template (ใช้ฟอร์ม "ส่วนราชการ / ที่ / วันที่ / เรื่อง / เรียน / สิ่งที่ส่งมาด้วย / ข้อความ").
* **A-GOV-2 (medium, currency):** อ้างอิงระเบียบสารบรรณ 2526 + "(และฉบับแก้ไขเพิ่มเติม)" — มีการแก้ไขสำคัญใน **พ.ศ. 2548** และ **พ.ศ. 2564 (e-Saraban)**. ควรระบุเลขให้ชัด.
  * *Action:* เปลี่ยนวลีเป็น "ระเบียบสำนักนายกฯ ว่าด้วยงานสารบรรณ พ.ศ. 2526 (แก้ไขเพิ่มเติม ฉบับที่ 2 พ.ศ. 2548, ฉบับที่ 4 พ.ศ. 2564 — e-Saraban)" หรือ phrasing เทียบเท่า + เพิ่มย่อหน้าสั้นๆ ว่าหน่วยงานปัจจุบันส่วนใหญ่ส่งหนังสือผ่าน e-Saraban ของ DGA.

### thai-pdpa

* **A-PDPA-1 (highest priority across audit, currency):** ขาดอ้างอิง **Standard Contractual Clauses templates** ที่ PDPC เผยแพร่ในปี 2566-2567 สำหรับ cross-border transfer ที่ปลายทางไม่ได้อยู่ใน adequacy list.
  * *Action:* เพิ่มย่อหน้า "SCC: ใช้เมื่อปลายทางไม่อยู่ในประกาศ adequacy" พร้อมลิงก์ PDPC.
* **A-PDPA-2 (currency):** ไม่กล่าวถึง **DPIA (Data Protection Impact Assessment)** ทั้งที่ PDPC ออกแนวทางเป็นการเฉพาะ.
  * *Action:* เพิ่ม section "DPIA: เมื่อใดต้องทำ" — high-risk processing, large-scale sensitive data, systematic monitoring.
* **A-PDPA-3 (gap):** ไม่ครอบคลุมกฎ **เฉพาะภาคส่วน** ที่ override / เสริม PDPA — ธนาคาร (BOT), โทรคมนาคม (NBTC), สุขภาพ (สำนักงานหลักประกันสุขภาพ / กรมการแพทย์).
  * *Action:* เพิ่ม section "Sectoral overrides" — ถ้าใน sector นี้ ต้องอ่านกฎภาคส่วนคู่กัน.
* **A-PDPA-4 (gap):** ไม่กล่าวถึง **Joint controller agreement** — กรณีองค์กรสองแห่งเป็น controller ร่วมของชุดข้อมูลเดียวกัน.
  * *Action:* เพิ่มย่อหน้าสั้นๆ พร้อมเงื่อนไขใน Sec 6 / 39.
* *Note (medium, interpretation):* การตีความ Sec 20 (เด็ก) ในกลุ่มอายุ 10-20 มีแนวทาง PDPC ออกตามมา. ปัจจุบันใน skill บอกถูกตามตัวบท แต่ถ้าจะใช้ในงานจริง ควรอ้าง PDPC notification ฉบับล่าสุด.

### thai-text-processing

* **A-TXT-1 (medium, gap):** ไม่กล่าวถึง **LLM tokenizer behaviour on Thai** — ภาษาไทยถูก tokenize ละเอียดมากใน GPT-4 / Claude (3-5x ต้นทุนของอังกฤษเทียบเท่าตัวอักษร). dev ที่ build LLM app เป็นภาษาไทยต้องรู้.
  * *Action:* เพิ่ม subsection "LLM tokenization" ประมาณ 8-10 บรรทัด.
* **A-TXT-2 (low, gap):** ไม่กล่าวถึง **TIS-620 legacy encoding** — ยังพบใน DB เก่า, CSV จากระบบเก่า, PDF ราชการ.
  * *Action:* เพิ่มหนึ่งย่อหน้าในส่วน Unicode normalization.

### thai-id-validate

* **A-ID-1 (medium, gap):** ไม่อธิบาย **first-digit prefix convention** ของบัตรประชาชนไทย (1 = เกิดในไทยหลัง 2527, 3 = ก่อน 2527, 8 = ต่างชาติ PR, ฯลฯ). บางครั้ง validator ต้องแยกระหว่างคนไทยและต่างชาติ.
  * *Action:* เพิ่มตาราง first-digit prefix.
* **A-ID-2 (medium, gap):** ไม่กล่าวถึง **Slip QR verification** (มาตรฐาน bill payment QR หลังปี 2566 จากสมาคมธนาคาร) — ต่างจาก PromptPay แต่มักถูกสับสน.
  * *Action:* เพิ่ม section "Slip QR vs PromptPay QR" สั้นๆ.

### thai-date-format

* **A-DATE-1 (high, gap):** ไม่มีการแปลง **ปฏิทินจันทรคติไทย** (ขึ้น/แรม X ค่ำ เดือน Y). วันสำคัญทางพุทธทุกวันใช้ปฏิทินจันทรคติ — การแปลงไปสุริยคติเป็นงานที่ขอเข้ามาจริง.
  * *Action:* เพิ่ม section "วันจันทรคติ" พร้อมตัวอย่าง (วันวิสาขบูชา → ขึ้น 15 ค่ำ เดือน 6).
* **A-DATE-2 (medium, gap):** ไม่กล่าวถึง **time format conventions** — "08.30 น." (ใช้จุดและ "น." ตามท้าย) เป็น Thai convention. ใน skill ไม่มี.
  * *Action:* เพิ่มตาราง time format (08.30 น. / 8 โมงครึ่ง / 08:30).
* *Note (low):* ไม่กล่าวถึง **ปีนักษัตร** (12-animal cycle) — น่ามีในอนาคต.

### thai-address

* **A-ADDR-1 (high, code):** Parser ตัดชื่อถนนแบบ single-token. ถนนหลายคำเช่น "พระราม 9", "เพชรเกษม 21", "สุขุมวิท 26" ถูกตัดผิด. ระบุใน README's Known Limitations อยู่แล้ว แต่ยังไม่ได้แก้.
  * *Action:* extend regex ของ `road` ให้รับ multi-token (allow numeric suffix + ตัวอักษรไทย).
* **A-ADDR-2 (medium, gap):** ไม่ครอบคลุม **คอนโด / อาคาร / หมู่บ้าน** prefix (ห้อง / ชั้น / ตึก) ซึ่งอยู่ในที่อยู่ทันสมัยส่วนใหญ่.
  * *Action:* เพิ่ม field `unit` (ห้อง/ชั้น/ตึก/อาคาร) ลงใน parser output.
* **A-ADDR-3 (low, canonical):** ไม่ระบุว่า **"กรุงเทพมหานคร" เป็นรูปแบบทางการเดียว** — "บางกอก", "กรุงเทพฯ", "กทม." พบใน raw data.
  * *Action:* เพิ่ม note ใน "Bangkok vs provincial" + normalize ใน parser ให้ map → "กรุงเทพมหานคร".

### thai-invoice

* **A-INV-1 (high, gap):** ไม่ครอบคลุม **e-Tax invoice (เอกสารอิเล็กทรอนิกส์)** — กรมสรรพากรบังคับสำหรับธุรกิจที่รายได้เกินเกณฑ์ (ปัจจุบัน 30 ล้านบาท/ปี). หลายคนต้องใช้.
  * *Action:* เพิ่ม section "e-Tax invoice" พร้อมเงื่อนไขและช่องทาง (ระบบ e-Tax Invoice & e-Receipt ของกรมสรรพากร).
* **A-INV-2 (medium, gap):** ไม่อ้าง **ภ.พ.30 (monthly VAT return)** — เป็นเอกสาร VAT ที่ยื่นบ่อยที่สุด.
  * *Action:* เพิ่มในส่วน VAT (2-3 บรรทัด).
* **A-INV-3 (medium, gap):** ไม่กล่าวถึง **ภ.ง.ด.1 (salary WHT)** — เป็นเอกสาร payroll ที่นิติบุคคลต้องยื่นทุกเดือน.
  * *Action:* เพิ่มในตาราง WHT ที่มีอยู่.
* *Note (low):* **อากรแสตมป์ (Stamp Duty)** สำหรับสัญญาบางประเภท ก็ไม่อยู่ใน skill — อาจเป็น skill แยกในอนาคต.

## Theme ที่ตัดข้าม skill / Cross-cutting themes

1. **Digital-government coverage บางในสาม skill** — `thai-government-form` (e-Saraban), `thai-invoice` (e-Tax), `thai-id-validate` (Slip QR). ทั้งสามตัวเหมือนสะท้อนยุค paper-form ทั้งที่รัฐไทยขยับเข้าสู่ดิจิทัลอย่างจริงจังตั้งแต่ปี 2563.
2. **thai-pdpa มีความเสี่ยง currency สูงสุด** — ครอบคลุมตัวบทกฎหมายดี แต่ขาดประกาศ PDPC สองปีล่าสุด. ใครพึ่งพา skill นี้ในการตรวจ compliance ปี 2569 อาจพลาดข้อบังคับสำคัญ.
3. **Skill เรื่องชีวิตประจำวัน (resume / festival-card) ขาดเหตุการณ์สำคัญในวงจรชีวิตคนไทย** — ทหาร, อุปสมบท, ขึ้นบ้านใหม่, สมัครราชการ.
4. **ปฏิทินจันทรคติยังไม่ครอบคลุม** — `thai-date-format` แปลง พ.ศ.↔ค.ศ. แต่ไม่แตะ ขึ้น/แรม ซึ่ง `thai-festival-card` และ `thai-government-form` ต้องการโดยปริยาย.
5. **Trigger description ทั้ง 12 ตัวทำได้ดี** — ไม่ต้องแก้ใน round นี้.
